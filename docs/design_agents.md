对于某些存在"通用执行者干不好、但价值密度够高"的位置，需要判断是否值得单开专项 Agent 。

## 建议单开的专项Agent：

**一、图书管理员 Agent（你架构里已经提到的，但值得展开）**

你在 3.6 节已经设计了 Librarian Agent，但目前描述偏"善后清理"。我认为它的职责应该比你写的更重。它其实是整个系统记忆质量的守门人，具体该做三件事：任务结束后的降噪摘要提炼（你已经写了）；知识库的冲突检测与合并仲裁（比如两次任务产出了对同一问题的矛盾结论，它要标记出来而不是默默覆盖）；以及周期性的"记忆衰减"——对长期未被引用的知识条目降权或归档，防止检索层被过时信息污染。这个 Agent 的记忆权限应该是系统里唯一的例外：它需要 Staged-Knowledge 写权限，但每次写入必须生成变更日志供人类审计。

**二、外部会话摄入 Agent（Ingestion Specialist）**

你在 3.3 节提到了"外部 AI 会话摄入"，这个功能的复杂度被严重低估了。把 ChatGPT、Claude Web、Gemini 的聊天记录导入并转化为"高质量上下文资产"，绝不是简单的格式转换。它需要做的事情包括：识别对话中的有效结论与无效发散（大量脑暴对话里可能只有 20% 是有价值的）；提取隐含的决策点和被否决的方案（这些"路没走通"的记录对后续任务同样重要）；以及把非结构化的对话流重组为结构化的知识条目，带上来源标注和置信度。

这件事通用执行者做不好，因为它要求 Agent 对"什么算有价值的知识"有领域判断力，而且输入格式高度异构（不同平台的导出格式完全不同）。它天然适合做成专项 Agent，输入是一段外部对话记录，输出是结构化的知识条目集合，边界极其清晰。记忆权限应该是 Staged-Knowledge——它产出的条目进入暂存区，等图书管理员或人类审核后才正式入库。

**三、文献解析 Agent（Literature Specialist）**

你的目标用户是学术研究人员，而学术工作流里最耗时的环节之一是文献处理。这个 Agent 的职责不是"读论文然后总结"——通用执行者就能做。它的高价值场景是：从一批 PDF 里提取结构化的方法论对比表（比如"这五篇论文分别用了什么数据集、什么模型、什么评估指标，结果如何"）；识别文献之间的引用关系和观点冲突；以及把提取的信息与你知识库里已有的笔记（Obsidian）做关联匹配。

这就是你架构里"领域 RAG 扩展包"的典型消费者。学术 PDF 的切块策略和普通文档完全不同——摘要、方法、实验、结论需要按语义段落切而不是按 token 数切，表格和图表需要单独提取，公式需要保留上下文。这个 Agent 挂载一个学术文献的 Domain RAG Package，配合专门的解析 Skill，能把文献处理从"每篇手动读"变成"批量结构化入库，按需精读"。

**四、质量审查 Agent（Reviewer / Validator）**

你的架构在 Agent Taxonomy 里定义了"审查者"这个角色，但没有具体展开。我认为至少需要一个通用的质量审查 Agent，它的职责是在关键节点对其他 Agent 的产出做独立校验。比如：代码 Agent 生成的脚本是否真的能跑通、输出是否符合预期；文献 Agent 提取的数据是否与原文一致（这个问题比想象中严重，LLM 提取表格数据时经常出错）；绘图 Agent 输出的图是否符合期刊投稿规范。

这个 Agent 的设计要点是**必须无状态**——它不持有任何记忆，每次调用只接收"待审查的工件 + 审查标准"，输出"通过/不通过 + 具体问题列表"。这是你架构里防止"隐式全局记忆污染"原则的天然落地点。它也不应该有修复能力——发现问题后打回给对应的执行者或专项 Agent 去改，职责分离要干净。

**五、编排策略顾问 Agent（Meta-Optimizer）**

这个对应你 3.6 节的"元优化与自我迭代"。但我把它从一个抽象概念收窄成一个具体的专项 Agent：它定期（比如每周）扫描 Event Log 和已归档的任务轨迹，做三件事——识别反复出现的任务模式并提议新的 Workflow 模板；发现频繁失败或需要人工干预的环节并提议 Skill 优化；以及分析模型路由的实际表现（比如某类任务路由给模型 A 的成功率是否显著低于模型 B）并建议路由策略调整。

它的输出不是直接的系统变更，而是"提案"——交给人类决定是否采纳。记忆权限是只读的，只能读取 Event Log 和 Artifact Store，不能写入任何东西。这保证了它是一个纯粹的建议者，不会意外改变系统行为。

**总结一下优先级**

按 Phase 1 到 Phase 3 的节奏排列：

Phase 1 应该引入的是**质量审查 Agent**，因为它无状态、边界最清晰、实现最简单，而且从第一天起就能防止通用执行者的产出带着错误流入下游。Phase 2 随着知识融合建设引入**文献解析 Agent** 和**外部会话摄入 Agent**，它们是领域 RAG 扩展包的核心消费者。**图书管理员 Agent** 也在 Phase 2，因为知识开始大量入库后才需要守门人。**编排策略顾问** 放在 Phase 3，因为它需要足够多的历史数据才能产出有意义的建议——系统跑了几个月、积累了几百条任务轨迹之后，它才有料可分析。

对于科研绘图 Agent， 可以 Phase 1 先做 Skill，Phase 2 再升级为专项 Agent，和文献解析 Agent 同期落地比较合理。
