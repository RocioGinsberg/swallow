# 知识摄入与RAG架构设计 (Knowledge Ingestion & RAG Design)

## 1. 核心愿景：超越传统RAG，迈向 Graph RAG 与全局理解
在处理复杂的软件工程和系统架构任务时，标准的“切块与嵌入”（Chunk & Embed）RAG模式往往会失效。传统的RAG丢失了局部代码与全局架构之间的上下文联系，导致“语义碎片化”；在面对需要多步推理或依赖链路追踪的查询时，单纯的向量相似度检索常常返回相关但无用的信息片段。

为了解决这一痛点，我们不仅构建**混合检索图谱 (Hybrid Knowledge Graph)**，更引入了 **Graph RAG (图检索增强生成)** 的高级范式。
*   **混合图谱结构**：将图结构的确定性关系（引用、继承、属于）与向量检索的模糊语义匹配相结合，使系统能够顺藤摸瓜，提供具有深度上下文和逻辑链条的检索。
*   **系统级实体提取与全局问答 (Graph RAG)**：Graph RAG 不仅仅是图遍历，它是对海量语料进行系统性的实体、关系和声明（Claims）提取。通过社区检测（Community Detection）算法将知识图谱聚类，并为每个社区生成层次化的摘要。这种架构能够完美解答全局性（Global）问题，例如：“所有这些架构文档中的共同主题是什么？”或“整个分布式系统的容灾策略是如何演进的？”，而传统的向量检索面对这类需要跨文档“纵览全局”的问题时常常束手无策。

## 2. 领域专属的数据摄入与切块策略 (Domain-Specific RAG Packages)
不同类型的数据包含不同的语义结构，必须采用特定的提取与扩展策略：

*   **代码仓库 (Code & AST)**：
    *   *策略*：基于抽象语法树 (AST) 进行解析，而不是简单的文本长度切分。
    *   *处理方式*：按类、函数、接口级别进行切块；提取函数签名、Docstring、输入输出依赖；保留跨文件的导入和调用链。
*   **长篇文档与学术PDF (Legal/Academic PDFs)**：
    *   *策略*：层级化切块 (Hierarchical Chunking)。
    *   *处理方式*：利用版面分析还原文档树结构（标题 -> 章节 -> 段落）；保留跨页的引用、图表说明和脚注信息；生成章节级别的摘要（Summary）以建立“父块”，并保留具体细节作为“子块”。
*   **事件日志与追踪 (Event Logs)**：
    *   *策略*：基于时间线 (Timeline-based) 与因果链切分。
    *   *处理方式*：提取时间戳、Trace ID 和状态变更 (State Mutations)；将连续的事件流聚合为单个“会话”或“事务”切块，保留系统状态在时间维度上的演变脉络。

## 3. 外部AI会话摄入与任务交接 (External AI Handoff)
在现实工作流中，用户常常已经在 ChatGPT 或 Claude Web 界面中进行了初步的探索和讨论。系统需要无缝继承这些外部上下文，防止“从头开始”的知识流失。

*   **工作流摄入**：支持导入 ChatGPT/Claude 的对话记录（JSON/Markdown/分享链接）。
*   **结构化解析**：系统利用 LLM 对原始漫谈进行提纯，自动提取三大核心要素：
    1.  **上下文 (Context)**：当前系统的状态、讨论的业务背景。
    2.  **约束 (Constraints)**：技术选型限制、非功能性要求、不能改动的历史代码。
    3.  **目标 (Goals)**：最终需要交付的产物或解决的核心问题。
*   **生成权威资产**：提纯后的信息被转化为标准化的**“任务交接单” (Task Handoff Note)**。这个交接单将作为知识库中的一个核心（Canonical）资产，指导后续所有 Agent 的规划与执行操作。

## 4. 增强检索流水线 (Enhanced Retrieval Pipeline)
为保证大规模知识图谱的高效精准召回，我们设计了四个阶段的检索增强流水线（作为底层基础检索能力）：

1.  **查询重写与意图识别 (Query Rewrite)**：
    *   使用快速模型将用户的模糊提问进行拆解和扩展，补全缺失的代词，生成同义词，或将复杂问题分解为多个子查询。
2.  **混合检索 (Hybrid Search)**：
    *   *向量检索 (Dense)*：捕获深层语义相似性。
    *   *关键字检索 (BM25/Sparse)*：精准匹配代码变量名、特定的报错信息或专有术语。
    *   *局部图谱遍历 (Local Graph Traversal)*：基于命中的节点，沿知识图谱边缘提取其直接依赖或父级上下文。
3.  **智能重排 (Cohere Rerank / Cross-Encoder)**：
    *   将混合检索召回的 Top-K 候选块进行打分和重新排序。通过更复杂的注意力机制计算查询与候选块的精确相关度，剔除冗余和伪相关内容。
4.  **大模型综合与生成 (LLM Synthesis)**：
    *   将经过排序的高质量上下文注入到生成模型的 Prompt 中。强制模型在输出时附带信息来源标识（Citations），确保生成的每一行代码和架构建议都有迹可循、可溯源。

## 5. Agentic RAG (动态/智能体检索)：从静态流水线到自主决策
上述的“增强检索流水线”依然是一个“固定流程”（Fixed Pipeline）。面对极其复杂、需要探底的系统工程问题时，我们引入更高维度的 **Agentic RAG**，将检索行为从“流水线”升级为智能体的“决策问题”。

在 Agentic RAG 范式下，Agent 会利用大模型的逻辑推理能力（Reasoning）主动管理信息获取过程：

1.  **动态路由与工具选择 (Routing & Tool Use)**：
    *   Agent 首先评估用户问题，决定**是否**需要进行外部检索（若当前上下文已足够则直接回答）。
    *   在决定检索后，自主选择最合适的**检索工具**：需要局部细节时调用 Vector DB；需要解答宏观全局问题时调用 Graph RAG；需要精确统计验证时调用 SQL/代码执行器；甚至直接调用 Web Search 获取最新外部依赖文档。
2.  **多跳推理与迭代检索 (Multi-hop Reasoning)**：
    *   打破“一次提问、一次检索”的限制。Agent 会执行 `检索 -> 阅读 -> 意识到信息缺失 -> 构建新查询 -> 再次检索` 的循环。
    *   *示例场景*：Agent 检索到报错日志提示某个接口 `AuthService.verify()` 失败，它会自主发起第二次检索，定向查找 `AuthService` 的源码实现，甚至第三次检索查找该文件相关的历史改动记录。
3.  **文档相关性自我反思 (Self-Reflection & Correction)**：
    *   如果初步检索到的文档片段无法回答问题，Agent 不会强行产生“幻觉”回答，而是会触发反思机制（Reflection），判断当前召回质量不足，主动调整搜索关键词、扩大范围或改变搜索策略以重新尝试。
