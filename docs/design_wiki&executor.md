# AI 调度系统设计调整总结（Wiki / Runtime / Subagents）

> 基于本轮讨论整理，面向项目长期规划。  
> 主题覆盖：LLM Wiki、RAG 定位、通用/专项 Agent 分工、自研 Runtime、厂商原生 Agent 的使用边界、Subagents 设计思路。

---

## 1. 当前方向的核心判断

本轮讨论的总体结论是：

- **项目应继续走“统一调度系统”路线，而不是退化成多模型聊天器。**
- **三家通用 Agent（Codex / Claude / Gemini）适合进入系统，但不应成为系统本身。**
- **系统的真正核心应是你自己的调度器、知识层、Runtime 和任务编排能力。**
- **厂商原生 Agent 更适合作为外部执行器（executor），而不是总控层。**
- **专项 Agent 应大量使用便宜模型 / 本地模型 / 低成本 API，但其角色必须由系统规则严格约束。**

一句话概括：

> 应构建一个“你自己的统一调度系统”，再把强模型、原生 Agent、专项 Agent 都吸纳为可路由的执行资源。

## 2. LLM Wiki 的定位调整

### 2.1 为什么值得加入

对于 AI 调度系统，单纯的 RAG 只能解决“找得到资料”的问题，但很难稳定解决：

- 历史决策之间的关系
- 模块职责边界
- 某条规则为什么存在
- 文档、代码、设计是否一致
- 长期项目认知的沉淀与复用

因此，引入 **LLM Wiki** 是有价值的。

### 2.2 它不替代 RAG

新的定位建议是：

- **RAG**：原始资料检索层 / 证据层
- **LLM Wiki**：结构化知识层 / 认知层
- **调度器**：决定何时查 Wiki、何时回源到 RAG、何时直接执行

所以不应理解为“有了 Wiki 就不需要 RAG”，而应理解为：

> Wiki 是建立在 RAG 之上的高层知识组织层。

### 2.3 建议的知识分层

建议至少分为三层：

1. **原始资料层**  
   代码、README、AGENTS、PR、Issue、笔记、日志、外部文档。

2. **RAG 检索层**  
   负责切块、索引、召回、重排、回源。

3. **Wiki 认知层**  
   负责沉淀：
   - 术语
   - 模块职责
   - 决策记录
   - 工作流规则
   - 阶段总结
   - 风险与技术债

### 2.4 Wiki 的写入原则

Wiki 不应成为自由生成的“第二知识幻觉层”。必须遵循：

- 高价值、可复用、相对稳定的信息才进入 Wiki
- 每条结论尽量带来源指针（PR / 文档 / 代码 / 决策记录）
- 需要复核后再写回长期知识层
- 不允许所有 Agent 随意直接改写 Wiki

### 2.5 建议的第一批 Wiki 页面

长期规划中，可优先考虑：

- 系统目标
- 调度器职责
- 通用 Agent 分工
- 专项 Agent 分类
- 权限分级规则
- Runtime 结构说明
- 决策记录（ADR）
- 当前技术债与待验证设计

## 3. 通用 Agent 的重新定位

本轮讨论中，三家通用 Agent 的最佳定位不是“谁最强”，而是“谁最难被替代”。

### 3.1 Codex

最不可替代之处：**稳健施工**

适合：
- 明确实现任务
- 修 bug
- 补测试
- 小到中型重构
- 输出可审查 diff

建议角色：
- 默认工程 executor
- implementation agent

### 3.2 Claude

最不可替代之处：**规划与审查质量**

适合：
- 方案比较
- 任务拆解
- 风险识别
- review 与纠偏
- 大改动前的约束建模

建议角色：
- planner
- reviewer
- architect-support agent

### 3.3 Gemini

最不可替代之处：**长上下文吸收与知识整合**

适合：
- 大仓库理解
- 多文档汇总
- 长历史上下文整理
- Wiki 草稿与知识底稿生成
- 文档与实现一致性检查

建议角色：
- knowledge agent
- documentation agent
- consistency agent

### 3.4 压缩后的职责总结

- **Codex**：把任务做出来
- **Claude**：把任务想清楚
- **Gemini**：把上下文吃进去并组织起来

## 4. 专项 Agent 的定位调整

### 4.1 重要原则

专项 Agent 不应与通用 Agent 混为一谈。

建议明确：

> 通用 Agent 负责判断与编排，专项 Agent 负责标准化产出。

### 4.2 专项 Agent 的模型策略

专项 Agent 应优先使用：

- 便宜 API
- GLM / Qwen / MiniMax 等性价比模型
- 本地模型
- 规则 + 小模型混合

因为专项任务通常：

- 边界清晰
- 输入输出格式固定
- 容易做 schema 校验
- 可批处理
- 对成本敏感

### 4.3 专项 Agent 适合承担的任务

- OCR 后字段抽取
- 财务票据核对
- 文本分类
- 论文摘要抽取
- 术语表整理
- 翻译初稿
- 邮件分类
- 网页抓取与标准化
- 数据清洗
- 报表生成

### 4.4 不应交给专项 Agent 的任务

不建议让便宜模型专项 Agent 直接做：

- 架构路线选择
- 阶段规划
- 高层优先级判断
- 长期知识库共识修改
- 关键 merge / 决策裁决

这些任务仍应升级给通用 Agent 或人工。

## 5. 工程 / 研究 / 日常三大任务域的适配思路

本轮讨论明确了：

三家通用 Agent 不应只被看作“工程 Agent”。更合理的理解是：它们分别代表三种高阶认知模式。

### 5.1 Claude

在研究与日常任务中的定位：**分析 / 判断者**

可承担：
- 研究问题拆解
- 提纲设计
- 路线比较
- 利弊分析
- 风险审查
- 重要文本审阅

### 5.2 Gemini

在研究与日常任务中的定位：**知识整合者**

可承担：
- 多篇论文整合
- 长笔记整理
- 概念图谱与 Wiki 草稿
- 项目历史归档
- 长对话整理

### 5.3 Codex

在研究与日常任务中的定位：**可执行化者**

可承担：
- 数据处理脚本
- 绘图代码
- 自动化工具
- 批处理脚本
- 文件 / 表格操作自动化

### 5.4 结论

系统不应按“工程 / 非工程”去分模型，而应按“认知角色”来分配职责。

## 6. 模型、Agent 框架、调度器三者关系的澄清

本轮讨论中，关于“角色到底由模型决定还是框架决定”的结论如下：

### 6.1 三层结构

一个 Agent 可理解为三层叠加：

1. **基础模型层**  
   决定能力上限、风格偏好、成本、速度、上下文能力。

2. **Agent Runtime 层**  
   决定它如何调用工具、如何读写文件、如何执行循环、如何管理上下文与状态。

3. **调度与权限层**  
   决定它在系统里该做什么、不该做什么、做到哪一步、何时升级。

### 6.2 关键判断

- 模型决定“它能做多好”
- Runtime 决定“它怎么做”
- 调度器决定“它该不该做、何时做、做到哪一步”

### 6.3 对系统设计的启发

因此，在你的系统中：

> Agent 的角色应该主要由你自己的调度系统和权限框架决定，而不是由模型品牌直接决定。

也就是说：

- 先定义角色（规划者、施工者、知识整合者、抽取器、翻译器等）
- 再为角色选 Runtime
- 最后为 Runtime 绑定模型

角色先于模型。

## 7. 产品路线：统一页面 vs 厂商原生 Agent

### 7.1 原始分歧

需要在两种路线之间选择：

- **方案一**：做统一页面，直接调用三家模型 / 多模型 API
- **方案二**：直接依赖 Codex / Claude Code / Gemini CLI 作为主要工作入口

### 7.2 本轮建议

更推荐以 **方案一为主**，即：

- 前台只有一个统一调度界面
- 后台统一管理任务、记忆、知识层、权限、成本、执行器路由

而厂商原生 Agent：

- 作为外部 executor 接入
- 在特定任务（尤其工程任务）中调用
- 不承担总控层职责

### 7.3 建议的双轨结构

系统内部应支持两类路由：

1. **模型路由**  
   直接调模型 API

2. **执行器路由**  
   调厂商原生 Agent Runtime（如 Codex / Claude Code / Gemini CLI）

### 7.4 原则

- 你的系统负责“组织学”
- 厂商 Agent 负责“特定任务上的成熟执行能力”

这能避免：

- 系统退化成“多模型聊天器”
- 或被厂商原生页面绑架为“外部 Agent 启动器”

## 8. Runtime 路线调整

你倾向方案一后，核心问题变为：Runtime 要不要自建，以及参考什么。

### 8.1 总体建议

建议自建一个 **轻量、任务域无关、可替换的 Runtime v0**，并借鉴公开开源项目的思路，而不是依赖泄露代码或深度绑定某一家厂商产品。

### 8.2 可借鉴对象

讨论中认为有借鉴价值的方向包括：

- OpenClaw：常驻 runtime、多入口、gateway / daemon 思路
- Hermes：技能注册、经验沉淀、长期记忆、自我改进方向
- 厂商公开 Agent 产品中体现出的 loop / tool-use / executor 思路

### 8.3 不建议碰的内容

不建议把 Claude Code 泄露源码作为实现依据。理由包括：

- 非正常公开的专有代码
- 合规风险高
- 可能混入恶意内容
- 不适合成为未来开源 / 商用系统的基础

可以参考公开报道对行业趋势的总结，但不应触碰泄露实现本身。

### 8.4 建议的 Runtime v0 模块

建议 Runtime 至少包括以下模块：

#### Router
- 任务分类
- 权限等级判断
- 选择模型执行器或 Agent 执行器

#### Planner
- 将用户请求转成任务卡
- 识别可并行子任务
- 决定查 Wiki / 查 RAG / 直接执行

#### Executor Interface
统一抽象三类执行器：
- `ModelExecutor`
- `ToolExecutor`
- `AgentExecutor`

#### Memory Stack
- session memory
- task memory
- project wiki
- raw RAG store

#### Review Gate
- schema 校验
- 置信度判断
- 冲突检测
- 是否升级到强模型 / 人工

### 8.5 Runtime 的核心价值

这套 Runtime 的意义在于：

- 让通用任务、研究任务、专项任务都能进同一框架
- 让厂商原生 Agent 成为可插拔资源，而不是总控中心
- 为后续长期记忆、Wiki、自定义专项 Agent 提供统一底座

## 9. Subagents 设计调整

### 9.1 厂商原生 Subagents 的价值

你观察到厂商原生 Agent 也有 Subagent 能力，这一点确实有价值。

但建议的理解方式是：

> 可以借鉴 Subagent 的思想，但系统核心不应建立在某一家原生 Subagent 特性上。

### 9.2 为什么不能深度依赖厂商原生 Subagent

原因包括：

- 三家实现并不对齐
- 多数仍偏向 coding 场景
- 对研究 / 日常任务不一定适配
- 你对成本、权限、知识写回的控制会减弱
- 会降低系统级一致性与可替换性

### 9.3 最优姿势：两层 Subagent 机制

建议将 Subagents 分成两层：

#### 外层：你自己的平台级 Subagents
由你的调度器控制，负责：
- 拆任务
- 并行执行
- 汇总结果
- 冲突检测
- 复核升级

这是系统核心。

#### 内层：executor-native Subagents
某些外部执行器内部自带的子代理机制，例如某些厂商原生 Agent。  
这只能作为局部增强能力。

### 9.4 平台级 Subagents 的适用场景

#### 研究类
- 多篇论文并行摘要
- 多理论流派并行梳理
- 多数据源并行采集
- 时间线 / 术语表 / 方法表分工生成

#### 工程类
- 多模块影响分析
- 文档一致性检查
- 测试缺口排查
- 多方案并行比较

#### 专项批处理类
- OCR 后字段抽取
- 邮件分类
- 网页抓取
- 财务单据校验

### 9.5 建议新增的抽象

长期规划中，可在 Runtime 中显式引入：

- `SubtaskSpec`
- `ExecutorCapability`
- `supports_native_subagents`
- `max_parallelism`
- `allowed_scopes`
- `cost_profile`

这样就能把“你自己的多代理编排”和“外部执行器原生多代理能力”统一纳入一个框架。

## 10. 推荐的总架构抽象

综合本轮讨论，一个更清晰的总架构如下：

```text
User / UI
  -> Router
  -> Planner
  -> Knowledge Layer (Wiki / Memory / RAG)
  -> Subtask Orchestrator
  -> Executors
       - Model Executors
       - Tool Executors
       - Agent Executors
  -> Review Gate
  -> Memory / Wiki Writeback
```

### 10.1 各层职责

#### Router
- 判断任务域（工程 / 研究 / 日常 / 批处理）
- 判断复杂度与权限等级

#### Planner
- 生成任务卡
- 切分 subtasks
- 决定是否并行

#### Knowledge Layer
- 提供原始检索
- 提供高层认知摘要
- 支持长期项目记忆

#### Subtask Orchestrator
- 创建平台级 subagents
- 编排串行 / 并行 / 汇聚复核

#### Executors
- 模型 API
- 工具调用
- 厂商原生 Agent

#### Review Gate
- 保证输出结构、置信度与升级路径

#### Memory / Wiki Writeback
- 仅将高价值、复核通过的内容沉淀为长期知识

## 11. 协作范式建议

### 11.1 默认范式

> 通用 Agent 编排，专项 Agent 干活。

即：

- Claude / Gemini / Codex 负责高阶认知环节
- 专项 Agent 负责标准化工种
- 调度器负责串起来

### 11.2 推荐链路（工程）

Gemini 读全局  
-> Claude 做判断与拆解  
-> Codex 施工  
-> Claude / Gemini 复核  
-> 你做 PR 决策

### 11.3 推荐链路（研究）

Gemini 生成知识底稿 / Wiki 草稿  
-> Claude 做研究问题拆解与结构判断  
-> 专项 Agent 做筛选 / 摘要 / 翻译 / 整理  
-> Codex 做数据处理与可执行工具  
-> 结果沉淀到 Wiki / Memory

### 11.4 推荐链路（日常）

Router 判断任务  
-> Claude 处理复杂判断  
-> Gemini 整理长期资料与计划  
-> Codex 或工具执行自动化  
-> 专项 Agent 批量处理低认知任务

## 12. 应写入项目长期规划的关键规则

建议在长期规划中明确以下制度：

### 12.1 权限等级

可定义：

- L0：抽取 / 分类 / 改写
- L1：生成候选答案
- L2：汇总并给出建议
- L3：制定执行方案
- L4：修改长期知识 / 触发关键操作

便宜专项 Agent 主要停留在 L0-L1。  
通用 Agent 主要在 L2-L3。  
L4 尽量需要严格审批或人工确认。

### 12.2 任务卡制度

每个 Agent 都尽量通过明确任务卡工作，而不是开放式长聊。

任务卡建议包含：

- 目标
- 输入
- 输出 schema
- 约束
- 置信度阈值
- 升级条件
- 是否允许联网 / 写文件 / 执行命令

### 12.3 失败升级机制

以下情况应自动升级到通用 Agent 或人工：

- 低置信度
- schema 不合法
- 多结果冲突
- 涉及高权限操作
- 涉及长期知识写回
- 涉及架构 / 路线决策

### 12.4 知识写回机制

不是所有结果都写入长期记忆。  
只有：

- 高复用
- 相对稳定
- 经过复核
- 有来源支撑

的信息，才写入 Wiki / 长期 Memory。

## 13. 最终路线总结

综合本轮讨论，设计调整方向如下：

### 13.1 产品层
- 采用统一调度界面
- 不让用户直接频繁切换厂商原生界面

### 13.2 架构层
- 自建轻量 Runtime v0
- 以 Router / Planner / Executors / Review / Memory 为核心

### 13.3 知识层
- 保留 RAG 作为证据层
- 引入 LLM Wiki 作为认知层
- 建立 Memory / Wiki / RAG 的分层关系

### 13.4 Agent 层
- 三家通用 Agent 作为高阶认知层
- 大量专项 Agent 作为低成本执行层
- 明确“通用负责判断，专项负责产出”

### 13.5 执行器层
- 支持模型 API
- 支持工具执行器
- 支持厂商原生 Agent 执行器
- 厂商原生 Agent 仅作为可插拔 executor

### 13.6 多代理层
- 自建平台级 Subagents
- 厂商原生 Subagents 只作为局部增强

### 13.7 治理层
- 明确权限等级
- 引入任务卡制度
- 引入 Review Gate
- 引入知识写回规则

---

## 14. 最核心的长期定位

如果把整个讨论最终压缩成一句话：

> 这个项目的正确长期形态，不是“接很多模型”，也不是“包装别人的 Agent”，而是构建一个拥有自己知识层、Runtime、任务编排和治理规则的 AI 调度系统，再把模型、厂商 Agent 和专项 Agent 都吸纳为可路由资源。

这意味着你的系统核心竞争力最终将来自：

- 知识组织能力
- 调度与权限能力
- 多 Agent 协作结构
- 任务治理与长期记忆能力

而不是某一个单独模型的强弱。
